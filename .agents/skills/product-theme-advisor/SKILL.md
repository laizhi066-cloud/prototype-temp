# 产品主题顾问

先定设计方向，再写页面。主题不是装饰，而是产品类型、用户场景和信息密度的结果。

## 触发场景

- 产品经理第一次描述一个新产品或新页面。
- 产品需求从后台切到 C 端 Web、H5、活动页或数据看板。
- 用户指出页面“AI 味重”“太模板”“不像真实产品”。
- 需要修改登录页、首页、导航、主色、字体、卡片、表格或整体视觉气质。

## 工作流

1. 从产品描述提取行业、目标用户、使用设备、核心任务、业务严肃度和信息密度。
2. 给出 2 到 3 个主题方向，但只推荐 1 个默认方案。
3. 说明推荐理由：为什么适合这个产品，而不是因为好看。
4. 把推荐落到 `src/config/productProfile.js` 的产品上下文和页面文案。
5. 后台 / SaaS / CRM 默认使用克制、清晰、信息密度适中的设计；不要做营销页式大视觉。

## 推荐格式

写代码前用中文输出：

- 产品类型：
- 目标用户：
- 推荐主题：
- 配色方向：
- 字体与密度：
- 登录页方向：
- 不采用的风格：

## 设计规则

- 后台管理、SaaS、CRM、运营系统：优先安静、专业、低干扰、数据清晰。
- C 端 Web：优先品牌识别、转化路径和内容层级。
- H5 移动端：优先触控友好、单列流程、轻量动效和清晰主操作。
- 活动页：可以更强视觉，但必须服务报名、转化或传播。
- 数据看板：优先指标层级、筛选效率、图表可读性和异常提示。

## 禁止事项

- 不要默认套蓝白大卡片和抽象背景图。
- 不要在登录页展示“AI”“原型”“脚手架”“示例”等产品外身份。
- 不要为了酷炫使用装饰性渐变、漂浮光斑或无业务含义插画。
- 不要把主题只理解成主色；还要包含布局密度、字体、图标、卡片、表格和状态表达。
- 不要问产品经理“要什么颜色”，先根据产品做推荐。

## 验收

- 第一轮实现说明里有主题推荐和不采用风格。
- 登录页第一眼像具体业务产品，不像模板说明页。
- 运行界面文案来自具体产品场景。
- 颜色、间距、圆角、阴影和组件风格保持一致。
