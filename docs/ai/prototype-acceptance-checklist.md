# 原型验收清单

产品同事评审原型时，不需要看代码，优先看这些点。

## 1. 产品表达

- 页面里是否出现真实业务名称，而不是“原型”“脚手架”“AI”“示例”？
- 登录页是否只展示 logo、系统名称、简单系统描述和登录表单？
- 页面名称、按钮文案、状态文案是否像真实产品？
- 目标用户和核心场景是否能从流程中看出来？

## 2. 项目类型

- 后台管理 / SaaS / CRM 是否有登录、头部、树状侧边栏、内容区、用户菜单和退出登录？
- C 端 Web 是否避免后台侧边栏？
- H5 是否使用移动端布局、单列内容和触控友好控件？
- 数据看板是否有筛选区、指标卡、图表区和明细表？

## 3. 核心流程

- 核心流程是否能点击走通？
- 新增、编辑、删除、筛选、分页、详情是否符合业务？
- 危险操作是否有确认？
- 操作完成后是否有成功反馈？

## 4. 交互状态

- 是否有加载态？
- 是否有空状态？
- 是否有错误提示？
- 表单是否有校验？
- 失败后是否给出恢复方式，例如重试、修改字段或返回？

## 5. 数据模拟

- 数据是否通过 mock API 模拟前后端？
- 刷新页面后关键数据是否能保留？
- 是否有重置本地数据的入口？
- 模拟数据是否像真实业务，而不是随便凑的占位内容？

## 6. 视觉和可用性

- 字号是否以 12px 为紧凑基准，但正文和按钮仍然可读可点？
- 按钮、表格、弹窗、表单是否风格一致？
- 桌面和移动端是否没有文字溢出、遮挡或横向滚动？
- 页面是否避免明显的 AI 模板感，例如无意义大渐变、抽象装饰、登录前业务指标？

## 7. 交付前让 AI 做的检查

```text
使用 prototype-review。
请按 docs/ai/prototype-acceptance-checklist.md 审查当前原型。
发现问题后直接修复，并按改动类型选择必要验证。
完成后请创建中文 commit，并告诉我 commit 信息。
```

## 8. 验证不用过度

- 文档、规则、prompt 修改不用跑全量测试。
- 页面样式修改重点看截图。
- 路由、登录、导航、核心流程修改再跑 smoke。
- 产品同事只需要知道验证是否通过，不需要阅读完整测试日志。
