# 原型评审

像产品经理评审原型一样检查。

## 检查清单

- 产品目标和目标用户是否能从流程中看出来。
- 项目类型和采用的默认骨架是否匹配。
- 后台页面是否包含登录、头部、侧边栏、内容区和 Element Plus 组件。
- C 端和 H5 页面是否没有误用后台侧边栏。
- 核心流程是否可以点击走通。
- 加载、空状态、错误、校验、成功反馈和危险操作确认是否完整。
- 布局和组件是否复用，而不是每个页面重写一套。
- 数据是否通过 `src/mock/mockApi.js` 流转。
- 桌面和移动端视口下文字是否溢出或遮挡。
- 是否避免了巨型单文件输出和无关框架改动。
- 如果原型要交给开发，是否已经生成 PRD、用户故事、字段 / 状态说明、验收用例和开发任务拆分。

## 分级验证

- 文档、规则、prompt、skills 修改：检查文件存在、链接路径和关键规则，不需要运行构建或 smoke。
- 产品文案、配置、主题 token 修改：运行构建；涉及视觉时补截图。
- 页面 UI、布局、路由、登录、导航、核心业务流程修改：运行构建和 smoke。
- PRD、交付包、验收清单修改：检查文档结构完整；除非同时改运行页面，否则不跑 smoke。

## 完成规则

声明完成前按改动类型选择最轻量但足够的验证。最终回复面向产品经理，只说明验证结果，不贴大量测试日志。

如果本轮存在有效文件修改，完成验证后必须创建中文 Git commit，并在最终回复中说明 commit 信息。
